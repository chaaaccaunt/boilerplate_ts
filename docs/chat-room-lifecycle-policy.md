# Политика жизненного цикла chat rooms

## Назначение

Документ описывает пользовательское удаление, выход из комнат, передачу закрытых комнат системному владельцу и административный hard delete.

## Статусы комнат

`chat_rooms.status` описывает жизненный цикл комнаты:

- `active` — комната доступна обычным пользователям по правилам доступа;
- `archived_by_owner` — group room закрыта владельцем и передана системному пользователю;
- `orphaned` — private room больше не имеет активных участников и передана системному пользователю.

Обычные пользовательские списки комнат должны показывать только `active` rooms.

## Участники комнат

`chat_room_members.leftAt` фиксирует выход пользователя из комнаты.
Строка membership не удаляется физически при обычном выходе пользователя, чтобы сохранить историю участия.

Активным участником считается запись, у которой:

- `deletedAt` отсутствует;
- `leftAt` равен `NULL`.

Проверка доступа к непубличной комнате должна учитывать только активных участников.

## Public rooms

`public` room является системной обязательной комнатой.

Обычные пользователи не могут:

- удалять public room;
- покидать public room;
- переводить public room в архивный статус.

Отсутствие обязательной public room является controlled error, а не поводом выбирать fallback-комнату.

## Group rooms

`group` room имеет владельца.
В текущей модели владельцем считается `chat_rooms.createdByUserUid`, пока комната активна.

Владелец group room может:

- изменить название комнаты;
- удалить комнату пользовательским действием.

Пользовательское удаление group room не выполняет hard delete.
При удалении владельцем:

- все активные участники получают `leftAt`;
- комната получает статус `archived_by_owner`;
- `createdByUserUid` меняется на системного пользователя;
- `archivedAt` фиксирует время закрытия;
- сообщения и связи вложений остаются в базе для будущего админского просмотра.

Обычный участник group room, который не является владельцем, может покинуть комнату.
После выхода комната исчезает из его пользовательского списка, но остается доступной другим активным участникам.

## Private rooms

`private` room закрывается для каждого пользователя отдельно.

Если один участник выходит из private room:

- его membership получает `leftAt`;
- для второго активного участника комната остается доступной.

Если второй участник тоже выходит:

- комната получает статус `orphaned`;
- `createdByUserUid` меняется на системного пользователя;
- `archivedAt` фиксирует время закрытия;
- комната скрывается из обычных пользовательских списков.

## Системный пользователь

Закрытые комнаты передаются системному пользователю:

```text
00000000-0000-4000-8000-000000000202
```

Этот пользователь создается migration/seed layer и используется только как технический владелец закрытых комнат.
Он не должен использоваться как обычная учетная запись пользователя.

## Admin dashboard и hard delete

Административный hard delete не выполняется в пользовательском chat UI и доступен только через admin dashboard.

Admin dashboard:

- показывать комнаты со статусами `archived_by_owner` и `orphaned`;
- позволять администратору открыть историю комнаты;
- выполнять hard delete только после явного подтверждения администратора.

Hard delete удаляет:

- связи `chat_message_files`;
- сообщения `chat_messages`;
- участников `chat_room_members`;
- саму комнату `chat_rooms`.

Hard delete не удаляет физические файлы и записи `stored_files`.
Файл принадлежит пользователю и может использоваться вне конкретной комнаты, поэтому удаление файлов должно выполняться отдельным явно описанным сценарием.
