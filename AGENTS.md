Правила для агента

0. Политика выполнения задач

- Агент обязан выполнять только явно запрошенный объем работы.
- Если действие не было явно запрошено пользователем, агент считает его запрещенным, даже если оно кажется логичным продолжением задачи.
- Создание сущности не означает ее автоматическое подключение. Подключение, интеграция и изменение вызывающего кода выполняются только по отдельному явному запросу.
- Если задача неоднозначна, затрагивает архитектуру или может изменить поведение соседнего кода, агент обязан сначала задать уточняющий вопрос.
- Если агент видит ошибку вне запрошенной области, он должен сообщить о ней, но не исправлять без отдельного разрешения.
- Перед изменением существующих файлов агент обязан кратко указать, какие файлы будут изменены и зачем.
- `AGENTS.md` должен содержать только общие правила работы агента и высокоуровневые архитектурные ограничения.
- Детальные проектные conventions, примеры кода, порядок объявления сущностей и инструкции по конкретным подсистемам должны храниться в `./docs`.
- Вся проектная документация, включая файлы в `./docs` и новые инструкции для агентов, должна быть написана на русском языке.
- Перед выполнением задачи агент обязан просмотреть содержимое `./docs` и прочитать релевантные документы для затрагиваемой области, включая backend и frontend правила.
- Frontend и backend должны синхронизироваться через сериализованные JSON DTO/contracts из shared-слоя, а не через backend runtime classes, ORM models или Node-only types.

1. Архитектура и проектные conventions

- Backend, frontend и shared-contract conventions описаны в `./docs`.
- Перед работой над подсистемой агент обязан просмотреть `./docs` и прочитать релевантные документы для затрагиваемой области.

2. Политика объявления типов

- Агент обязан выносить domain types и interfaces в package-local `@types` согласно правилам из проектной документации.
- Подробные backend, frontend, shared-contract и type declaration conventions описаны внутри `./docs`.
