Правила для агента

0. Политика выполнения задач

- КРИТИЧЕСКОЕ ПРАВИЛО: агент не имеет права изменять файлы, запускать интеграции или менять поведение проекта без прямого повелительного запроса пользователя.
- Вопросы пользователя вида "есть ли", "посмотри", "как думаешь", "нужно ли", "можешь ли", "не так ли" являются запросом на анализ или ответ, а не разрешением на изменение файлов.
- Если пользователь не написал явно "сделай", "исправь", "добавь", "убери", "перенеси", "создай", "подключи" или другой прямой приказ на действие, агент обязан ответить текстом и остановиться.
- После анализа агент может предложить изменение, но не должен выполнять его до отдельного явного подтверждения пользователя.
- Агент обязан выполнять только явно запрошенный объем работы.
- Если действие не было явно запрошено пользователем, агент считает его запрещенным, даже если оно кажется логичным продолжением задачи.
- Создание сущности не означает ее автоматическое подключение. Подключение, интеграция и изменение вызывающего кода выполняются только по отдельному явному запросу.
- Если задача неоднозначна, затрагивает архитектуру или может изменить поведение соседнего кода, агент обязан сначала задать уточняющий вопрос.
- Если агент видит ошибку вне запрошенной области, он должен сообщить о ней, но не исправлять без отдельного разрешения.
- Перед изменением существующих файлов агент обязан кратко указать, какие файлы будут изменены и зачем.
- `AGENTS.md` должен содержать только общие правила работы агента и высокоуровневые архитектурные ограничения.
- Детальные проектные conventions, примеры кода, порядок объявления сущностей и инструкции по конкретным подсистемам должны храниться в `./docs`.
- Вся проектная документация, включая файлы в `./docs` и новые инструкции для агентов, должна быть написана на русском языке.
- Все пользовательские сообщения приложения, тексты в верстке, labels, placeholders, кнопки, ошибки, уведомления, console messages и иные человекочитаемые строки должны быть написаны на русском языке, если пользователь явно не попросил другой язык.
- Агент не должен сокращать нейминги в коде, документации, URL, routes, API endpoints, DTO, переменных, методах, файлах и директориях без явной причины и отдельного согласования.
- URL, routes и API endpoints должны иметь полные, читаемые и однозначные названия. Сокращения вроде `auth`, `cfg`, `usr`, `pwd`, `msg` не используются, если пользователь явно не попросил такой формат.
- Все создаваемые и изменяемые текстовые файлы проекта должны сохраняться в кодировке UTF-8.
- Проект должен быть строгим к данным, конфигурации, типам и контрактам. Агент не должен добавлять fallback-значения для обязательных полей, env-переменных, API contracts, DTO, state или runtime-зависимостей.
- Fallback разрешен только для явно optional-поля или optional-настройки, где отсутствие значения является ожидаемым и описанным поведением.
- Если обязательное значение отсутствует, код должен явно падать с понятной ошибкой или возвращать контролируемую ошибку, а не подставлять молчаливое значение по умолчанию.
- Перед выполнением задачи агент обязан просмотреть содержимое `./docs` и прочитать релевантные документы для затрагиваемой области, включая backend и frontend правила.
- Frontend и backend должны синхронизироваться через сериализованные JSON DTO/contracts из shared-слоя, а не через backend runtime classes, ORM models или Node-only types.

1. Архитектура и проектные conventions

- Backend, frontend и shared-contract conventions описаны в `./docs`.
- Корневая структура fullstack boilerplate описана в `./docs/repository-architecture-conventions.md`.
- Перед работой над подсистемой агент обязан просмотреть `./docs` и прочитать релевантные документы для затрагиваемой области.

2. Политика объявления типов

- Агент обязан выносить domain types и interfaces в package-local `@types` согласно правилам из проектной документации.
- Подробные backend, frontend, shared-contract и type declaration conventions описаны внутри `./docs`.
